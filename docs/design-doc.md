
# Experiment Management Platform — Design Doc (v0.1)

## 1. なぜこのシステムを作るのか

LLM・CV・評価パイプラインを含む現代の実験では、
「実験を回す」ことよりも、**「どのデータで、どの条件のときに、何が良くなり、何が壊れたのか」**を追跡することが圧倒的に難しい。

多くの実験管理ツール（MLflow等）は：

* 実験 = run 単位
* メトリクス = run に1つか2つ
* データ（サンプル） = 外部のどこか

というモデルのため、

> 「この1つのデータ（case）が、条件Aでは失敗し、条件Bでは成功する」

という最も重要な問いに答えられない。

このシステムの目的は、
**“case を軸に実験を観測・比較できる実験管理”** を成立させることである。

---

## 2. この設計の核心

このシステムは、実験を次の3層で分離して扱う。

```
Project
  └ Batch (実験条件の固定)
        └ Case (データ単位の結果)
```

ここで重要なのは：

* **Batchは管理単位**
* **Caseは比較・分析の単位**

という役割分担である。

---

## 3. なぜ「同一Case」を定義する必要があるのか

実験で本当に知りたいのは：

> この入力データは、なぜこの条件では失敗し、別の条件では成功したのか？

これを可能にするには、「同じ入力」を **確実に同一視**できなければならない。

そのため本システムでは：

```
同一Case = (dataset_version_id, sample_id)
```

を厳密な同一性キーとして採用する。

* datasetが違えば別物（原則）
* sample_idはクライアントが必ず付与（無ければ自動生成）
* dataset_versionもクライアントが管理（ユーザ指定を支援）

これにより：

* 同じ画像・同じ文章・同じ行データが
* 別のBatch（別条件）でどう振る舞ったか

を横断的に追跡できる。

---

## 4. なぜ「Batch一次」なのか

UI・検索・比較・ランキングの起点は **Batch** にする。

理由：

* 人間が考える単位は「この実験条件の結果」
* しかし最小単位はCaseであり、そのままでは量が多すぎる

Batchを一次にすることで：

* まず「どの実験が良いか」をBatchで見る
* 次に「なぜ良いか/悪いか」をCaseで掘る

という自然な分析フローが作れる。

---

## 5. なぜ Garbage Batch が必要か

実験は無限に増える。

すべてを一覧・比較・ランキングに出すと：

* UIが崩壊
* 集計が重くなる
* 意味のある結果が埋もれる

そこで本システムは：

> 「ユーザがチェックしなかった実験は価値が低い」

というヒューリスティックを採用する。

`unchecked_count >= 50` で Garbage にすることで：

* 放置された実験は自動的に表舞台から消える
* しかし削除ではなく、**復活可能**

この「捨てずに見えなくする」設計が運用を持続可能にする。

---

## 6. なぜ Parquet + RDB の二層構成なのか

### 問題

Caseは巨大で、数百万〜数億になる。
しかし：

* 再計算したい
* 再評価したい
* 新しい指標で見直したい

### 解決

Caseを2種類の情報に分離する。

| 種類                 | 役割                       | ストレージ  |
| -------------------- | -------------------------- | ----------- |
| **Facts**            | 再計算・再集計に必要な真実 | **Parquet** |
| **Metadata / Index** | 検索・絞り込み・操作       | **RDB**     |

Parquetは：

* 列指向
* 圧縮が効く
* Spark/DuckDBで再計算できる

RDBは：

* 一致・部分一致が速い
* タグ・条件での絞り込みが得意
* UIのレスポンスを担保できる

この分離により：

> 「DBで絞る → Parquetで再計算」

という強力な分析ループが成立する。

---

## 7. なぜスコアは Parquet に置くのか

スコアは：

* 定義が変わる
* しきい値が変わる
* 集計方法が変わる

つまり **再計算が前提のデータ** である。

そのため：

* `score_primary`
* `score_aux`

の2種類のスコアは **Facts として Parquet に保存**する。

RDBにあるスコアはあくまで：

> 「検索・並び替えのためのコピー」

であり、真実は Parquet 側にある。

---

## 8. なぜメタ情報は RDB なのか

タグ・モデル名・前処理・プロンプトバージョンなどは：

* 検索で多用される
* 後から増える
* 一致/部分一致が欲しい

これを Parquet に入れると：

* 列が爆発
* 部分一致が遅い
* スキーマ進化が地獄

RDBに置くことで：

* v0.1の「Gmail風検索」が成立
* 将来の検索UXが進化できる

---

## 9. なぜ横断検索が必須なのか

このシステムの価値はここにある。

横断検索により：

* 「この画像だけ、前処理ONだと壊れる」
* 「この文章だけ、プロンプトv12で改善した」
* 「このデータは回帰した」

という **改善と劣化の地図** を作れる。

これがなければ、
実験管理は「数値の比較」で終わる。

---

## 10. なぜ監査ログを最初から入れるのか

このシステムは将来：

* チーム
* 組織
* 課金

を前提とする。

「誰が何を見たか」は：

* セキュリティ
* ガバナンス
* 価値の源泉（誰が何を分析したか）

になる。

そのため：

* open project
* open batch
* （将来）case

をすべて **非同期で全件記録**する。

---

## 11. この設計で何ができるか

この設計により：

* 実験は Batch として管理され
* データは Case として横断的に比較され
* 改善と回帰が可視化され
* 再評価・再計算が可能で
* 実験が「資産」になる

---

## 結論

このシステムは、

> **「ログの保管庫」ではなく、「実験知識のエンジン」**

として設計されている。

ParquetとRDBの分離、Batch一次、Case横断、Garbage運用、監査ログはすべて、
その目的から必然的に導かれた設計である。
